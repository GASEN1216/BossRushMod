# BossRushMod 代码结构优化方向

> 目标：在不影响游戏内实际行为的前提下，降低后续新增模式、Boss、武器、NPC、奖励和 UI 的接入成本。
> 本文档不是一次性重写方案，而是后续具体优化时的方向文档和边界约束。

---

## 1. 优化原则

### 1.1 行为冻结

结构优化期间默认不改变以下内容：

- 物品 `TypeID`、Buff ID、存档 key、本地化 key。
- 场景名、地图配置值、刷新点坐标、商店库存、掉落概率。
- 玩家入口流程、入场条件、退场清理、奖励结算。
- 已有公开静态方法和外部调用入口，除非先保留兼容 wrapper。
- AssetBundle 目录、文件名、Prefab 命名规则。

每次重构都按“行为等价搬迁”处理：先建立新边界，再把旧逻辑原样迁移，最后用编译、静态 guard 和进游戏冒烟确认没有行为漂移。

### 1.2 小步迁移

不建议把 `ModBehaviour`、`Integration/` 或某个大玩法文件一次性拆完。更合适的节奏是：

1. 先抽公共生命周期和清理机制。
2. 再抽注册表和目录边界。
3. 最后按模式或系统逐个迁移。

每一轮只移动一个清晰责任域，保留旧入口方法名，让调用方先不变。

### 1.3 复用已有基础

当前代码里已经出现了几类正确方向，不应另起一套：

- `Common/Lifecycle/IBossRushRuntimeModule.cs`
- `Common/Lifecycle/BossRushRuntimeModuleHost.cs`
- `Common/Lifecycle/SceneRuntimeContext.cs`
- `Utilities/SceneRuntimeGate.cs`
- `Utilities/RuntimeScope.cs`
- `Utilities/RunScopedRegistry.cs`
- `Common/Stats/RuntimeStatModifierTracker.cs`
- `Integration/NPCs/Common/NPCModuleRegistry.cs`
- `Integration/ItemFactory.cs`
- `Integration/EquipmentFactory.cs`

后续结构优化应该围绕这些基础继续收束，而不是再增加新的并行生命周期体系。

## 2. 当前结构问题

### 2.1 `ModBehaviour` 仍是过重入口

现状：

- `ModBehaviour.cs` 约 5500 行。
- 大量 `partial class ModBehaviour` 分散在 `ModeD/`、`ModeE/`、`ModeF/`、`ZombieMode/`、`WavesArena/`、`Integration/`、`Utilities/`。
- `Awake`、`Start`、`Update`、`LateUpdate`、`OnSceneLoaded`、`OnDestroy` 仍直接串联大量系统。

影响：

- 新增功能时很容易继续往 `Update`、`OnSceneLoaded_Integration`、`OnDestroy_Integration` 里加分支。
- 生命周期入口顺序靠人工记忆维护。
- 一个系统的清理逻辑容易散在多个文件里。

方向：

- 让 `ModBehaviour` 逐步退化成“Unity 入口适配器”。
- 各子系统用 runtime module 接收生命周期事件。
- 旧的 `partial` 方法先保留 wrapper，再逐步把实现搬到模块类。

### 2.2 `Integration/` 目录过宽

现状：

- `Integration/` 是最大目录，生产 `.cs` 文件约 170 多个。
- 同时承载动态物品、装备、NPC、Boss、武器、建筑、商店、Wiki、好感度、重铸、场景注入。
- `Integration/BossRushIntegration.cs` 既做初始化，又做商店注入、场景分支、系统订阅、清理。

影响：

- 新系统默认会被塞进 `Integration/`，目录语义越来越模糊。
- 初始化顺序、注册顺序、资源加载顺序集中在一个文件里，后续很难局部验证。

方向：

- 保留 `Integration/` 作为“接原版游戏 API 的桥接层”，但不再让它承载玩法主逻辑。
- 内容系统应有自己的模块入口，例如 `RegisterItemConfigurators`、`RegisterEquipmentSystems`、`RegisterNpcModules`。
- `BossRushIntegration.cs` 最终只保留桥接、事件订阅和向模块 host 转发的薄层。

### 2.3 清理和生命周期模式不统一

现状：

- 有些系统靠静态缓存，有些系统靠实例字段，有些系统靠列表记录临时对象。
- 事件订阅、协程、临时 `GameObject`、运行时 `Modifier`、AssetBundle 缓存各自清理。
- `ZombieMode`、`ModeE`、`ModeF` 已经局部使用 `RunScopedRegistry` 和 `RuntimeStatModifierTracker`，但还没有成为全项目约定。

影响：

- 多次进出场景时容易出现重复订阅、静态引用残留、协程漏停、对象漏销毁。
- 新功能开发者需要模仿旧代码，但旧代码里有多套风格。

方向：

- 所有“本局/本场景临时状态”统一走 runtime scope 或 run-scoped records。
- 所有运行时 `Modifier` 统一走 `RuntimeStatModifierTracker`。
- 所有反向清理列表优先走 `RunScopedRegistry.ForEachReverse`。
- 所有场景级逻辑先过 `SceneRuntimeGate`。

### 2.4 注册逻辑仍然硬编码在主流程

现状：

- 物品配置器在 `InitializeDynamicItems_Integration` 中逐行注册。
- 装备和武器系统在 `Start_Integration` 中逐个初始化。
- 公共 NPC 已有 `NPCModuleRegistry`，但物品、装备能力、模式入口还没有同等级注册中心。
- `compile_official.bat` 仍需要手工列入新增 `.cs` 文件。

影响：

- 新增一个内容点通常要修改 3 到 5 个无关文件。
- 漏改编译清单会导致“源码存在但没进 DLL”。
- 注册顺序依赖人工维护，后续很容易发生初始化前后顺序问题。

方向：

- 短期继续保留 `compile_official.bat`，但配合 guard 保证所有生产源码被列入。
- 中期把物品、装备能力、模式入口拆成各自 registry。
- 长期再评估是否生成编译清单或迁移到更标准的项目文件，但这不是第一阶段目标。

### 2.5 大文件内部职责过多

典型文件：

- `Integration/DragonKing/DragonKingAbilityController.cs`
- `LootAndRewards/LootAndRewards.cs`
- `ZombieMode/ZombieModeRewards.cs`
- `Integration/WishFountain/WishFountainService.cs`
- `Integration/NPCs/Courier/StorageDepositService.cs`
- `ModeF/ModeFFortifications.cs`

影响：

- 单文件上下文过大，任何小改动都要读很多无关逻辑。
- UI、数据、运行时状态、特效、清理逻辑混在一起时，后续拓展容易出现副作用。

方向：

- 不做纯粹为了行数的拆分。
- 只在触碰对应功能时，按“数据模型 / 运行时控制 / UI 视图 / 资源加载 / 清理记录”拆。
- 拆分后旧公开方法保留在原文件或原类中转发，降低调用点 diff。

## 3. 目标结构

### 3.1 `ModBehaviour` 的目标职责

最终 `ModBehaviour` 只负责：

- 持有 `Instance`。
- 接收 Unity 生命周期。
- 初始化并调用 `BossRushRuntimeModuleHost`。
- 提供少量兼容性 wrapper，服务还没迁移的旧调用。
- 暴露确实需要被 Unity / 原版游戏 / Harmony 调用的入口。

不再直接承担：

- 具体模式状态机。
- 大量 UI 绘制逻辑。
- 物品配置注册表。
- NPC 生成规则。
- Boss 技能控制。
- 跨场景临时对象清理细节。

### 3.2 Runtime Module 分层

推荐把 runtime module 分成三类。

#### 全局模块

用于整个 Mod 生命周期都存在的系统：

- 成就系统
- 调试工具
- 本地化注入
- 动态物品和装备注册
- 公共 NPC 注册
- Wiki / 图鉴一类常驻数据

#### 场景模块

用于进入具体场景后才处理的系统：

- 基地商店注入
- 地图选择入口注入
- 公共 NPC 场景生成
- 婚礼建筑 / 许愿台建筑恢复
- 原版场景对象隔离和清理

#### 模式模块

用于某个玩法模式的完整生命周期：

- 标准 BossRush / 无间炼狱
- Mode D
- Mode E
- Mode F
- ZombieMode
- 后续 Mode G / Mode H / Mode I

每个模式模块至少要有：

- `CanStart` 或入口检测。
- `StartRun`。
- `OnSceneLoaded`。
- `Tick`。
- `EndRun`。
- `CleanupForSceneChange`。
- `CleanupForDestroy`。

### 3.3 推荐目录边界

短期不要大规模移动目录。建议在现有目录中增加“边界文件”：

```text
Common/
  Lifecycle/          # 已存在：runtime module host/context/interface
  Runtime/            # 建议：RuntimeScope、场景/局清理基础设施可逐步放这里
  Stats/              # 已存在：运行时 Modifier helper
  UI/                 # 建议：TMP 字体、通用按钮、轻量弹窗 helper

Config/
  BossRushMapCatalog.cs       # 建议：从 ModBehaviour 中迁移地图静态数据
  BossRushContentIds.cs       # 可选：未来把 ID 常量按内容域拆分

Integration/
  GameBridge/         # 可选：原版 API / 场景 / 商店桥接，不放玩法逻辑
  Items/              # 已存在：物品配置和使用逻辑
  NPCs/               # 已存在：NPC 模块
  Weapons/            # 可选：未来新武器过多时再拆，不急

ModeD/
ModeE/
ModeF/
ZombieMode/
WavesArena/
  *RuntimeModule.cs   # 建议：每个模式先增加一个模块外壳，内部调用现有方法
  *State.cs           # 已有或新增：模式状态容器
  *Cleanup.cs         # 复杂模式保留独立清理文件
```

注意：不要为了目录更漂亮而移动大量文件。先加薄模块，再逐步把实现搬进去。

## 4. 推荐迁移路线

### 阶段 0：建立行为基线

目的：先证明“当前是什么行为”，后续才知道有没有变。

建议工作：

- 固化 `compile_official.bat` 退出码和源文件覆盖 guard。
- 保留并扩展 `tests/OfficialCompileListFileExistenceGuard.py`。
- 保留并扩展 `tests/ArchitectureStructureGuard.py`。
- 为关键生命周期 token 加 guard：
  - `runtimeModuleHost.OnAwake`
  - `runtimeModuleHost.OnStart`
  - `runtimeModuleHost.OnUpdate`
  - `runtimeModuleHost.OnLateUpdate`
  - `runtimeModuleHost.OnSceneLoaded`
  - `runtimeModuleHost.OnDestroy`
  - `SceneRuntimeGate` 主菜单 / 加载屏 / 基地判断
- 写一份进游戏冒烟清单，至少覆盖：
  - 基地售货机注入
  - 地图选择
  - 标准 BossRush 开始 / 通关 / 离场
  - Mode D / Mode E / Mode F 入口和离场
  - ZombieMode 入口、波次、奖励、撤离
  - 公共 NPC 刷新和销毁
  - 新武器能力和常驻装备能力

阶段完成标准：

- 文档和 guard 存在。
- 正常编译通过。
- 无效路径编译能返回失败码。
- 进游戏冒烟结果有记录。

### 阶段 1：固定生命周期基础设施

目的：先让模块化入口稳定存在，但暂时不搬大逻辑。

建议工作：

- 确认 `BossRushRuntimeModuleHost` 的调用顺序固定。
- 明确模块注册发生在 `OnAwake` 分发之前。
- 增加一个轻量 no-op 模块作为模板，验证模块生命周期能被完整调用。
- 给 `RuntimeScope` 增加使用约定：
  - 场景级 scope：场景切换清。
  - 模式 run scope：模式结束清。
  - Mod scope：`OnDestroy` 清。
- 文档化模块异常策略：模块内部异常只打 warning，不阻断其它模块。

阶段完成标准：

- 没有任何玩法逻辑变化。
- `ModBehaviour` 生命周期仍能调用旧流程。
- 新模块系统只作为旁路存在。

### 阶段 2：迁移低风险横切系统

目的：优先搬“和玩法数值无关”的系统，验证模块化方式可靠。

建议优先级：

1. 调试 UI / DevMode 热键。
2. 成就 UI 快捷键和成就视图初始化。
3. 公共 NPC 场景生成的外壳。
4. 地图选择入口注入的外壳。
5. Wiki / 图鉴入口恢复一类数据系统。

迁移方式：

- 新建 `*RuntimeModule.cs`。
- 模块里先调用现有 `ModBehaviour` 私有方法或 internal wrapper。
- 保留旧方法和旧字段。
- 每搬一个系统就编译一次，并做对应冒烟。

阶段完成标准：

- `ModBehaviour.Update` 和 `LateUpdate` 中的横切逻辑明显减少。
- 调试、成就、NPC、地图选择行为与迁移前一致。

### 阶段 3：收束内容注册

目的：减少新增物品、装备、能力时对主流程的侵入。

建议新增：

- `Integration/Items/ItemContentRegistry.cs`
- `Integration/Equipment/EquipmentContentRegistry.cs` 或继续放在 `Integration/` 下的薄 registry
- `Integration/Systems/AbilitySystemRegistry.cs`，如目录过重可先不建新目录

物品注册目标：

```text
ItemContentRegistry.RegisterAllConfigurators()
  -> AwenCourierTokenConfig.RegisterConfigurator()
  -> ColdQuenchFluidConfig.RegisterConfigurator()
  -> ...

ItemFactory.LoadAllItems()
```

装备 / 武器系统目标：

```text
EquipmentContentRegistry.LoadAll()
  -> EquipmentFactory.LoadAllEquipment()
  -> DragonKingBossGunRuntime.InitializeRuntime()
  -> 各武器 TryConfigure

AbilitySystemRegistry.InitializeAll(owner)
  -> FlightTotem
  -> ReverseScale
  -> FenHuangHalberd
  -> Frostmourne
  -> PhantomWitchScythe
```

约束：

- 注册顺序必须与当前顺序一致。
- 不改变 `ItemFactory` 和 `EquipmentFactory` 对外行为。
- 不在第一轮引入自动反射扫描注册，避免初始化顺序变得不可控。

阶段完成标准：

- 新增一个普通物品只需要修改物品自身文件和 registry。
- `BossRushIntegration.cs` 中逐行注册明显减少。
- 编译清单 guard 继续覆盖新增文件。

### 阶段 4：模式生命周期模块化

目的：让新增模式不再继续扩大 `ModBehaviour` 的主循环。

建议顺序：

1. `WavesArenaRuntimeModule`，只封装标准 BossRush tick 和离场清理外壳。
2. `ModeDRuntimeModule`，Mode D 相对小，适合作为模式迁移模板。
3. `ModeERuntimeModule`，重点收束阵营缩放、商人、刷怪消耗品。
4. `ModeFRuntimeModule`，重点收束阶段状态机、悬赏、工事、撤离。
5. `ZombieModeRuntimeModule`，体量最大，最后迁移。

每个模式迁移先做“外壳模式”：

- 模块判断 active。
- 模块调用现有 `TickModeX`、`EndModeX`、`CleanupModeX`。
- 字段暂时仍留在 `ModBehaviour partial` 中。

外壳稳定后再逐步迁移字段：

- 先迁移纯状态对象。
- 再迁移清理记录。
- 最后迁移复杂业务方法。

阶段完成标准：

- `ModBehaviour.Update` 不再直接知道每个模式的 tick 细节。
- 每个模式都有独立入口、tick、清理位置。
- 新增 Mode G 时可以复制一个 runtime module 模板，而不是改主入口大段分支。

### 阶段 5：机会主义拆分大文件

目的：只在实际触碰功能时拆，不为拆而拆。

推荐拆分规则：

- UI 视图和业务服务分离。
- AssetBundle / prefab 加载与运行时控制分离。
- 配置表和技能执行分离。
- 清理记录和运行时状态集中放在 `*State` 或 `*Runtime` 文件。

示例方向：

- `LootAndRewards/LootAndRewards.cs`
  - 掉落池配置
  - 奖励箱生成
  - 胜利奖励
  - Mode E/F 共享 lootbox tracker
- `ZombieMode/ZombieModeRewards.cs`
  - 奖励目录
  - 奖励效果执行
  - 奖励 UI
  - 临时 NPC 服务
- `Integration/DragonKing/DragonKingAbilityController.cs`
  - 技能调度
  - 单个技能实现
  - 特效辅助
  - 清理记录

阶段完成标准：

- 每次拆分前后 compile 通过。
- 拆分只改变文件组织，不改变数值和调用时机。
- 旧调用点尽量少动。

## 5. 后续新增功能的推荐接入流程

### 5.1 新物品

目标流程：

1. 在 `Integration/Items/<ItemName>Config.cs` 定义 ID、bundle、prefab、本地化、tag。
2. 在同目录或相邻文件定义使用逻辑。
3. 在 `ItemContentRegistry.RegisterAllConfigurators()` 登记。
4. 如需售卖，在对应商店 registry 或场景模块登记。
5. 更新 `compile_official.bat`。
6. 跑 `OfficialCompileListFileExistenceGuard.py` 和编译。

不再建议：

- 直接继续把注册行塞进 `BossRushIntegration.cs`。
- 在物品配置文件里顺手处理不相关场景逻辑。

### 5.2 新武器 / 新装备能力

目标流程：

1. 资源仍遵循 `EquipmentFactory` 的命名规范。
2. 武器 ID、配置、Action、AbilityManager、Bootstrap 放在同一功能目录。
3. Bootstrap 登记到能力系统 registry。
4. 清理逻辑必须和初始化逻辑在同一模块中成对出现。

不再建议：

- 在 `Start_Integration` 里继续追加一组 `InitializeXxxSystem`。
- 把武器配置、技能执行、特效缓存、输入检测全部堆在一个文件里。

### 5.3 新 Boss

目标流程：

1. `Config`：数值、Prefab 名称、ID、阶段阈值。
2. `AssetManager`：资源加载、缓存、卸载。
3. `Boss`：生成、死亡、掉落、注册到 BossRush。
4. `AbilityController`：只负责技能调度。
5. `Abilities/` 或拆分文件：复杂技能单独文件。
6. `Cleanup`：随从、特效、事件、Modifier 必须可重复清。

不再建议：

- 一个 `AbilityController` 同时承担资源加载、技能实现、UI、清理和掉落。

### 5.4 新模式

目标流程：

1. 新建 `ModeX/ModeXModels.cs`。
2. 新建 `ModeX/ModeXRuntimeModule.cs`。
3. 入口物品或地图选择接入通过 registry。
4. 模式状态、刷怪、奖励、UI、清理拆文件。
5. 所有临时对象进入 mode run scope。
6. 所有场景切换都走模块自己的 cleanup。

不再建议：

- 在 `ModBehaviour.Update` 里直接加 `if (modeXActive) TickModeX(...)`。
- 在 `OnSceneLoaded_Integration` 里继续追加大段模式分支。

## 6. 安全重构清单

每次结构优化前：

- 确认当前工作树是否已有其它未提交改动。
- 记录本次只处理的责任域。
- 不同时修改玩法数值。
- 不同时做大规模 rename 和逻辑迁移。

每次结构优化中：

- 先新增新类和 wrapper。
- 再把旧逻辑原样搬过去。
- 保留旧入口直到调用点迁完。
- 每迁一个文件就更新 `compile_official.bat`。
- 新增或更新 guard，防止后续又绕回旧结构。

每次结构优化后：

- 运行 `python3 tests/OfficialCompileListFileExistenceGuard.py`。
- 运行相关架构 guard。
- Windows 下运行 `compile_official.bat`。
- 需要部署时运行 `test_bossrush_official.bat`。
- 做受影响系统的进游戏冒烟。

## 7. 第一批建议任务

### 任务 A：补全生命周期模块注册约定

目标：

- 明确模块在哪里注册。
- 确保 `OnAwake` 分发前已经注册完毕。

## 8. 不建议现在做的事

- 不建议立刻迁移到 `.csproj`。当前构建依赖本地游戏 DLL 和 Workshop Harmony 路径，先把编译清单 guard 做稳更实际。
- 不建议一次性移动整个 `Integration/` 目录。路径大变会制造巨大 diff，难以验证行为等价。
- 不建议把所有 `partial class ModBehaviour` 立刻改成独立 service。先用 runtime module 外壳降低主入口复杂度。
- 不建议在结构优化时顺手调数值、改掉落、改 UI 文案。这样会让行为回归难以定位。
- 不建议使用反射自动发现所有内容注册器作为第一步。显式 registry 虽然多一行，但顺序可控，适合当前 Mod 环境。

---

## 9. 最终判断标准

结构优化完成后，应该达到这些效果：

- 新增一个普通物品，不需要改 `ModBehaviour.cs`。
- 新增一个武器能力，不需要改 `Start_Integration` 主流程。
- 新增一个模式，不需要在 `Update` 和 `OnSceneLoaded_Integration` 中继续堆大分支。
- 每个模式都能清楚回答：什么时候开始、每帧做什么、什么时候清理、清理哪些对象。
- 每个临时对象、协程、事件订阅、运行时 Modifier 都有明确 owner。
- `compile_official.bat` 仍可用，并且 guard 能发现漏列源码。
- 玩家进游戏看到的行为与结构优化前一致。

这条路线的核心不是“把文件拆多”，而是把未来新增内容的改动范围收窄：新增功能主要改自己的目录和一个明确 registry，不再把所有系统继续压回 `ModBehaviour` 和 `BossRushIntegration`。
